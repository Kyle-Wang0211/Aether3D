## 2. 登峰造极新模块（建议纳入主文档）

### 2A. ERROR RECOVERY & DEGRADATION PROTOCOL

**动机**：正常路径与失败检测已定义完善，但失败后的恢复策略缺乏系统性设计。

| 错误码 / 场景 | 推荐恢复策略 |
|--------------|--------------|
| `AETHER_ERR_ARENA_EXHAUSTED` | 平台缩小 frame 预算或分帧；或返回 TIER_REDUCED 触发降级 |
| QG-1 跨平台 hash 不一致 | 生产 fallback：记录 telemetry、跳过该帧、触发回放诊断模式 |
| `AETHER_ERR_CANARY_CORRUPT` | 放弃当前 session；若有 checkpoint 则尝试从 checkpoint 恢复 |
| Merkle 损坏 | 优先抢救 Evidence > Merkle 证明 > TSDF mesh |
| 热降级后 | 每 N 帧采样 thermal；低于 THROTTLE_THRESHOLD 且持续稳定则自动升级回 TIER_FULL |

**契约**：
- Core 不执行恢复；仅返回 status。平台实现 `aether_recovery_matrix` 查找表。
- 数据抢救优先级：Evidence（可重算） > Merkle（审计） > TSDF（最重）

---

### 2B. OBSERVABILITY & TELEMETRY CONTRACT

**动机**：QG-10、Shard/Reveal 提到诊断，但缺系统性遥测契约。

| 计数器 | 含义 | 暴露方式 |
|--------|------|----------|
| `voxels_integrated_per_frame` | 每帧体素写入数 | 内存 ring buffer |
| `hash_computations_per_second` | 哈希调用频率 | 同上 |
| `arena_utilization_ratio` | Arena 使用率 [0,1] | 同上 |
| `canonical_float_ties_count` | ties-to-even 边界命中次数 | 可选，调试用 |

**约束**：Core 零 I/O。平台定期 `aether_telemetry_snapshot(buf, len)` 读取内存快照。  
**精度**：64-bit 原子计数器；采样间隔由平台控制（如每 100 帧）。

### 2C. WASM / WEB PLATFORM ADAPTATION LAYER

**动机**：HarmonyOS ArkUI + WebView、wasm32-unknown-unknown 目标需明确约束。

| 约束 | 应对 |
|------|------|
| 无 `__int128` | 64-bit 分拆；`_umul128` 风格软件乘 |
| Linear memory cap | 预分配固定 Arena；禁止动态扩展 |
| 无 mmap | 纯 heap；可选 SharedArrayBuffer 多线程 |
| WASM-SIMD | 替代 NEON 的 SIMD 路径；fallback 标量 |
| 线程模型 | Single-threaded 或 SharedArrayBuffer（需 CSP 头） |
| Emscripten vs WASI | 明确：Emscripten 用于浏览器；WASI 用于 Node/边缘 |

## 8. 独立调研与原创方案（非沿用审计建议）

> 以下为基于实际查阅与代码库分析的**原创发现与方案**，非机械沿用审计报告。  
> **已合并入主文档**：8.1 thermal 映射表 → Section 1.3；8.2+8.4 roundeven + Canonical Rounding → CanonicalFloat 小节；8.3 域前缀方案 → Section 3.5.6。

### 8.1 thermal_state_ordinal 跨平台不兼容（实测发现）

**调研结论**：
- **iOS** `ProcessInfo.ThermalState`：nominal=0, fair=1, serious=2, critical=3（4 级）
- **Android** `AThermalStatus`（NDK API 30+）：NONE=0, LIGHT=1, MODERATE=2, SEVERE=3, **CRITICAL=4**, EMERGENCY=5, SHUTDOWN=6（7 级）

**矛盾**：主文档 Section 1.3 写 "0=nominal, 1=fair, 2=serious, 3=critical" 为 **iOS 映射**。Android 的 CRITICAL 是 4，且 Android 有 EMERGENCY、SHUTDOWN 更高级别。

**原创方案**：Core 必须定义**平台无关的归一化 ordinal 0–3**，平台层负责映射：

| Core Ordinal | 含义 | iOS 映射 | Android 映射 |
|--------------|------|----------|--------------|
| 0 | NOMINAL | nominal | NONE |
| 1 | ELEVATED | fair | LIGHT, MODERATE |
| 2 | SEVERE | serious | SEVERE |
| 3 | CRITICAL | critical | CRITICAL, EMERGENCY, SHUTDOWN |

**契约**：`aether_evaluate_performance_tier(thermal_state_ordinal, battery_level_01)` 的 `thermal_state_ordinal` 必须为**已归一化的 0–3**，由平台在调用前完成映射。Core 不关心 iOS/Android 原始值。

### 8.2 roundevenf 编译器可用性（实测风险）

**调研结论**：C23 `roundeven`/`roundevenf` 已入标准（WG14 N2652），但 **Clang/GCC 在 NDK r26、Xcode 14–16 中的具体支持未在公开文档明确标注**。部分 C23 特性可能尚未全部落地。

**原创方案**：实现三层 fallback，在 `aether/canonicalize.h` 中：

```cpp
// 1. 若 __STDC_VERSION__ >= 202311L 且 roundevenf 可用 → 优先
// 2. 否则：fesetround(FE_TONEAREST); r = rintf(x);  (需单线程或线程局部)
// 3. 兜底：手动实现 ties-to-even（split float→ mantissa+exponent, round mantissa）
```

**Swift 端验证**：`Float.rounded(.toNearestOrEven)` 在 Swift 5+ 对应 IEEE 754 roundTiesToEven。建议增加 **golden fixture**：对 value×scale ∈ {0.5, -0.5, 1.5, 2.5, ...} 的边界值，Swift 与 C++ 输出必须逐位一致。

### 8.3 ReasonCode 全域 taxonomy（代码库实测）

**调研结论**：遍历 `Core/` 得出现有 reason 体系：

| 枚举 | 数量 | 域 | 文件 |
|------|------|-----|------|
| PrimaryReasonCode | 10 | PIZ/覆盖 | PrimaryReasonCode.swift |
| FailureReason | 17 | Job 流水线 | FailureReason.swift |
| RejectReason | 5 | 准入拒绝 | RejectReason.swift |
| GateInputInvalidReason | 17+ | 质量门 | GateInputInvalidReason.swift |
| MeshValidationFailureReason | 5 | Mesh 校验 | MeshValidationFailureReason.swift |
| CancelReason | 3 | 取消 | CancelReason.swift |
| Moderation ReasonCode (V2.6) | 5 | 审核 | 文档 |
| **合计** | **~58** | — | — |

**原创方案**：不合并为单一 58 项枚举（维护成本高），而是采用 **域前缀 + 统一转储**：

- Core 定义 `ReasonCodeDomain` 闭集：`PRC` | `FR` | `RR` | `GIR` | `MVF` | `CR` | `MOD` | `APPEAL` | `CHECKPOINT`
- 各域保留现有枚举；对外 API/遥测统一为 `{domain}_{code}` 字符串，如 `PRC_CAPTURE_OCCLUDED`、`FR_NETWORK_ERROR`
- `REASON_COMPATIBILITY.json` 已存在，约束 PRC 间组合；扩展其 schema 支持跨域引用

### 8.4 Canonical Rounding 可验证契约（原创）

**问题**：仅改 roundf→roundevenf 不够；需**可证明**的跨平台一致性。

**原创方案**：新增 "Canonical Rounding Specification" 一节：

1. **边界值 Fixture**：对每个 `Kind`，生成 `value * scale ∈ {k+0.5 : k∈Z}` 的输入，记录预期 canon 输出；CI 强制 Swift 与 C++ 结果一致。
2. **ULP 约束**：`canon` 与理想 ties-to-even 结果的误差为 0 ULP（整数输出无浮点尾差）。
3. **文档化**：主文档显式列出 "Ties-to-even 验证向量" 表，如 `(POSITION, 0.5/65536, 0)`、`(EVIDENCE, 0.5/16777216, 0)`。

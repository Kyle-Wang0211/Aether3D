## 1. 舍入 (roundf → roundevenf) — 多标准对齐

### 1.1 国际标准

| 标准 | 内容 | 引用 |
|------|------|------|
| **IEEE 754-2019** | roundTiesToEven（四舍六入五成双）为默认舍入模式；与金融计算、审计可追溯性对齐 | IEEE Std 754-2019 §4.3 |
| **IEC 60559:2020** | C 语言浮点与 IEEE 754 的绑定；C23 引入 roundeven/roundevenf | WG14 N2749、N2652 (TS 18661-5) |
| **ISO/IEC 9899:2023 (C23)** | §7.12.9.8 The roundeven functions；roundeven(±2.5)=±2.0（偶数）、roundeven(±3.5)=±4.0（偶数） | cppreference roundeven |
| **C23 Editorial N3242** | 2024 年对 "correctly rounded" 歧义修正，明确 ties-to-even 语义 | open-std.org |

### 1.2 国内标准

| 标准 | 内容 | 引用 |
|------|------|------|
| **GB/T 8170-2008** | 《数值修约规则与极限数值的表示和判定》；"四舍六入五成双"（奇进偶舍）与 IEEE 754 roundTiesToEven 等价 | 国标平台 std.samr.gov.cn |
| **应用场景** | 测试与计算得到的数值修约；金融、科学计算、工业测量 | GB/T 8170 适用范围 |

### 1.3 登峰造极实现契约

- **SSOT**: C23 `roundevenf` 为权威实现；pre-C23 使用 `rintf` + `fesetround(FE_TONEAREST)` 或手动 ties-to-even
- **Swift 对齐**: `Float.rounded(.toNearestOrEven)` 对应 IEEE 754 roundTiesToEven
- **可验证性**: Golden fixture 必须覆盖 value×scale ∈ {k+0.5 : k∈Z} 边界，Swift 与 C++ 逐位一致
- **标准引用**: 主文档显式引用 GB/T 8170、IEEE 754、C23 §7.12.9.8

## 2. 字节序 (static_assert 移除) — 跨平台可移植

### 2.1 工业实践

| 来源 | 方案 | 引用 |
|------|------|------|
| **Boost.Endian** | 显式转换 API：`native_to_little`、`conditional_reverse`；大端/小端平台均可编译 | boost.org Endian doc |
| **C++20 `<bit>`** | `std::endian`、`std::byteswap`；不依赖 native 端序假设 | C++20 Standard |
| **RFC 类实践** | 文档化 wire format（如 always little-endian），转换逻辑显式 | 网络协议惯例 |

### 2.2 登峰造极契约

- **禁止**: `static_assert(std::endian::native == std::endian::little)` 阻断大端构建
- **要求**: `aether_to_wire_le32()` 使用 `if constexpr` + `std::byteswap`；wire 格式固定为 LE
- **参考**: Boost.Endian 三种用法（conversion / buffer / arithmetic）；本方案采用 conversion 显式控制

## 5. 128 位整数 — 可移植实现

### 5.1 现状与提案

| 来源 | 内容 | 引用 |
|------|------|------|
| **P3140 (C++26)** | `std::int_least128_t` 提案；GCC/Clang/MSVC 已有 128-bit 扩展，库支持不完整 | open-std.org P3140 |
| **cppalliance/int128** | 可移植 128-bit；native 可用则用 builtin，否则软件 fallback | github.com/cppalliance/int128 |
| **MSVC** | 无 `__int128`；需 `_umul128` 或自定义 U128 类型 | MSVC docs |
| **WASM** | wasm32 无 128-bit；64-bit 分拆乘加 | WebAssembly spec |

### 5.2 登峰造极契约

- **优先**: 有 `__int128` 的编译器（GCC/Clang）使用 native
- **fallback**: MSVC → `_umul128` + 软件拼接；WASM → 64-bit split
- **可选依赖**: 评估 cppalliance/int128 作为 header-only 可移植层

## 6. NaN / StableLogistic — 数值稳定性

### 6.1 标准与实现

| 来源 | 内容 | 引用 |
|------|------|------|
| **IEEE 754** | NaN 比较均返回 false；NaN 传播；payload 传播实现不一 | IEEE 754-2019 |
| **scipy.special.expit** | 稳定 sigmoid：x≥0 用 1/(1+exp(-x))，x<0 用 exp(x)/(1+exp(x)) 防 overflow→NaN | Stack Overflow numerical-precision |
| **StableLogistic 特例** | 输入 NaN → 输出 0.5 为**特例**，与 Core 其它路径 NaN→AETHER_ERR_NAN_DETECTED 区分 | 主文档 V2.7 注释 |

### 6.2 登峰造极契约

- **特例显式化**: StableLogistic NaN→0.5 必须在文档和代码中注明为特例；其它路径禁止静默 NaN 传播
- **稳定性**: sigmoid 实现需采用 sign-branch 防 overflow

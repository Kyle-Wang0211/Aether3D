### 1.2 四个致命级Bug

**Bug 1：概率验证公式错误**

原始计划使用的公式：
```
k = ceil(ln(delta) / ln(1 - 1/N))
```
这是**有放回抽样**（二项分布）的公式。

Swift实际使用的公式（`VerificationMode.swift` 第75行）：
```swift
ceil(n * (1.0 - pow(delta, 1.0 / n)))
```
这是**无放回抽样**（超几何分布）的公式——对于文件验证场景是正确的（每个chunk只检查一次）。

两个公式在 N=10000, delta=0.001 时的结果对比：
- 错误公式：k ≈ 6907（抽样69%）
- 正确公式：k = 69（抽样0.69%）
- **差距100倍！** 错误公式会导致"概率验证"几乎等于全量验证，失去意义。

此外原始计划还额外加了一个 `ceil(sqrt(N))` 下限。Swift没有这个。已移除。

**Bug 2：域标签字节数全部写错**

| 域标签 | 原始计划写的 | 实际验证（Python len()） | BundleConstants.swift注释 |
|--------|------------|----------------------|------------------------|
| BUNDLE_HASH | 23 bytes | **22 bytes** | "// 22 bytes" |
| MANIFEST_HASH | 27 bytes | **26 bytes** | "// 26 bytes" |
| CONTEXT_HASH | 26 bytes | **25 bytes** | "// 25 bytes" |

原因：把Python源码中的 `\x00`（4个字符）误算为4个字节，实际只是1个字节（NUL）。

**Bug 3：路径遍历漏洞**

`upload_id` 是用户控制的输入，直接拼接到文件路径中：
```python
chunk_dir = Path(settings.upload_dir) / upload_id / "chunks"
```
如果攻击者传入 `upload_id = "../../etc"`，会导致：
```
storage/uploads/../../etc/chunks/ → /etc/chunks/
```
这是一个教科书级的目录遍历攻击。PR8的Swift代码有 `validateFileWithinBoundary()` 防护，但PR10完全没有。

**Bug 4：DB-文件写入顺序错误**

原始计划在 `upload_chunk()` 中先 `db.commit()` 再 `persist_chunk()`。如果 `persist_chunk()` 失败：
- DB记录说chunk存在（已提交）
- 文件实际不存在（写入失败）
- 组装时读取不到文件 → 崩溃

补丁修正为**文件先行，DB后行**：先写文件（可重试），再提交DB。如果DB提交失败，孤儿文件会被清理系统自动处理。

## 3. 思考过程与决策逻辑

### 3.1 为什么要做这么多补丁？

最初的PR10计划是"功能正确"的——它描述了正确的业务逻辑（三路管线、五层验证、三路去重、三层清理）。但它缺少了使PR1-8成为高质量代码的**所有元数据层**：

- PR8的 `ImmutableBundle.swift` 不仅仅是一个正确的实现——它是一个**自我文档化的安全合约**。每个关键决策都有SEAL FIX解释为什么这样做，每个不能随便改的地方都有GATE标记，每个保证都有INV-B编号。
- PR2的 `job_state.py` 不仅仅定义了状态机——它有宪法合约头声明了精确的计数（9状态、14转换、14失败原因），任何改动都会被断言检测到。

PR10如果没有这些元数据层，就像一座没有消防系统的摩天大楼——结构是对的，但缺少了让它在长期维护中保持安全的所有防护。

### 3.2 为什么概率公式错误如此严重？

这不仅仅是一个数学错误。它暴露了一个**认知差距**：

1. 原始公式 `ceil(ln(δ)/ln(1-1/N))` 来自于"连续抽样"的近似，假设每次抽样是独立的（有放回）。
2. 文件验证是"无放回"的——你不会验证同一个chunk两次。
3. 正确公式 `ceil(N*(1-pow(δ,1/N)))` 考虑了这一点，抽样量远小于错误公式。
4. 如果使用错误公式，N=10000时需要抽样6907个（69%），概率验证几乎失去意义。

Swift的实现（`VerificationMode.swift`第75行）使用了正确公式。服务器端必须完全匹配。

### 3.3 为什么要文件先行而非DB先行？

这是一个**失败模式分析**的结果：

| 失败场景 | 文件先行 | DB先行 |
|---------|---------|--------|
| 文件写入失败 | 无DB记录，无文件。干净状态。 | DB说chunk存在，文件缺失。**腐败状态。** |
| DB提交失败 | 孤儿文件存在磁盘。清理系统自动处理。 | N/A |
| 都成功 | 一致 | 一致 |

文件先行只有一种不一致状态（孤儿文件），是安全的（自动清理）。DB先行有一种不一致状态（幽灵记录），是危险的（组装时读取缺失文件会崩溃）。

### 3.4 为什么需要完整的4步fsync？

不完整的fsync在掉电时会导致数据丢失：

```
步骤1: 写入临时文件
步骤2: fsync(文件fd)         ← 确保数据到达磁盘
步骤3: rename(tmp → final)   ← 原子操作
步骤4: fsync(目录fd)         ← 确保目录条目（rename）持久化
```

- 如果缺少步骤2：掉电后文件可能是零长度（数据还在OS缓存中）
- 如果缺少步骤4：掉电后rename可能丢失（文件只在tmp路径下）

PostgreSQL WAL、LevelDB、SQLite WAL都使用完全相同的模式。这是POSIX下唯一可证明安全的持久化模式。

## 5. 我们的优势

### 5.1 技术优势

**1. 五层渐进验证引擎**

业界标准是1-2层验证（哈希对比+大小检查）。我们的五层引擎（结构→全文件→分块链→Merkle树→域分离bundleHash）提供了远超行业标准的完整性保证。关键创新是L1和L2的零额外I/O——它们是三路管线的副产品。

**2. 客户端-服务器字节一致的Merkle树**

极少有系统在客户端（Swift/iOS）和服务器（Python/Linux）之间维护字节级一致的Merkle树。这为未来的审计追踪、增量验证、链上证明提供了基础。

**3. 28个命名不变量**

这在开源项目中几乎是前所未有的。大多数项目即使有安全保证，也只是散落在注释中。INV-U编号系统使得每个保证可追溯、可审计、可引用。

**4. SEAL FIX / GATE系统**

这是一个**变更控制机制**，不仅仅是注释。GATE标记告诉未来的开发者"这段代码不能随便改，改之前需要RFC审批"。在快速迭代的团队中，这种机制比代码审查更有效地防止安全退化。

**5. 完整的错误分类（可重试/不可重试/安全违规）**

参考了PR2的 `FailureReason.is_retryable` 模式。客户端可以根据错误分类决定是否重试，而不是盲目重试所有失败——这减少了不必要的流量和用户等待。

### 5.2 架构优势

**1. 九个休眠能力的激活**

PR10不是从零开始——它是激活了PR1-8中已经定义但从未使用的9个能力。这意味着PR10的代码与现有代码库的衔接是**自然的**，不是强行嫁接。

**2. 零新依赖、零新端点、零新模型**

PR10只使用Python标准库（hashlib, hmac, os, pathlib, math, random）加上已有的项目依赖（sqlalchemy, fastapi）。不引入任何新的外部依赖、API端点或ORM模型。这意味着零额外维护负担。

**3. 宪法合约系统**

通过 `upload_contract_constants.py` 的计数断言（INVARIANT_COUNT=28, GATE_COUNT=9等），任何意外的增删都会在导入时触发断言失败。这是一个**编译时安全网**。

PR10的质量提升不是简单的"加注释"——它是将一个功能正确但缺少元数据层的实现，升级为一个**自我文档化、自我保护、自我审计**的安全合约系统。这与PR1-8建立的质量标准一致，确保Aether3D的服务器代码与客户端代码拥有同等的工程水准。
